---

## 1. 目的・背景

チームメンバーが「誰が」「何を」「どの程度の優先度で」抱えているかを瞬時に把握できる社内向けツールを構築する。これにより、リソース配分の最適化・業務の透明性向上・マネジメントコストの低減を実現する。

**設計方針:** 学習コストゼロ。入力ハードルを極限まで下げ、日常的に使い続けられるシンプルさを最優先とする。

---

## 2. スコープ

### 対象

- 社内チームメンバーのタスク管理
- マネージャーによるチーム全体のタスク俯瞰

### 対象外

- 外部公開・認証機能（ローカル運用のため不要）
- 工数管理・時間追跡
- 外部サービス連携（Slack / GitHub 等）

---

## 3. ユーザー定義

| ロール | 説明 | 主な操作 |
|:---|:---|:---|
| 一般メンバー | タスクの担当者 | タスクの登録・編集・ステータス変更・優先度更新 |
| マネージャー | チーム全体を管理する | 全メンバーのタスク俯瞰・ボトルネック発見・メンバー管理 |

## 4. 機能要件

### 4-1. メンバー管理

| # | 要件 | 優先度 |
|:---|:---|:---|
| M-01 | メンバーを新規登録できる（名前・アバター色を設定） | Must |
| M-02 | メンバー情報を編集できる | Must |
| M-03 | メンバーを削除できる（紐づくタスクは担当者なし状態に変更） | Must |

### 4-2. タスク管理（CRUD）

| # | 要件 | 優先度 |
|:---|:---|:---|
| T-01 | タスクを新規作成できる | Must |
| T-02 | タスク一覧を取得・表示できる | Must |
| T-03 | タスクを編集できる | Must |
| T-04 | タスクを削除できる | Must |
| T-05 | ステータスをドラッグ＆ドロップで変更できる | Must |
| T-06 | 優先度をワンクリックで変更できる | Should |
| T-07 | カンバン列内でのタスク並び順を保持できる（DnD後の位置を記憶する） | Must |

**タスクの入力項目:**

| フィールド | 型 | 必須 | 備考 |
|:---|:---|:---|:---|
| タスク名 | string | ○ | |
| 担当者 | user_id | ○ | ドロップダウン選択 |
| 優先度 | enum | ○ | 高 / 中 / 低 |
| ステータス | enum | ○ | 未着手 / 進行中 / 完了 |

| # | 要件 | 優先度 |
|:---|:---|:---|

## 5. 非機能要件

| 区分 | 要件 |

## 6. データモデル

シンプルさを保つためテーブルは 2 つのみとする。

### users

| カラム | 型 | 制約 | 説明 |
|:---|:---|:---|:---|
| id | TEXT | PK | cuid2 |
| name | TEXT | NOT NULL | 表示名 |
| avatar_color | TEXT | NOT NULL | UI アイコン色（HEX） |
| created_at | DATETIME | NOT NULL | 作成日時 |

### tasks

| カラム | 型 | 制約 | 説明 |
|:---|:---|:---|:---|
| id | TEXT | PK | cuid2 |
| title | TEXT | NOT NULL | タスク名 |
| user_id | TEXT | FK → users.id | 担当者（NULL = 担当者なし） |
| priority | INTEGER | NOT NULL | 1=高 / 2=中 / 3=低 |
| status | INTEGER | NOT NULL | 0=未着手 / 1=進行中 / 2=完了 |
| position | TEXT | NOT NULL | 並び順キー（Fractional Indexing）。同一ステータス内の表示順を保持する。 |
| created_at | DATETIME | NOT NULL | 作成日時 |
| updated_at | DATETIME | NOT NULL | 最終更新日時 |

**並び順の設計方針（T-07）:**  
`position` には `fractional-indexing` パッケージが生成する文字列キー（例: `"a0"`, `"a0V"`, `"a1"`）を格納する。DnD による並び替え・列移動時には **移動したタスク1件のみ** `position` を更新する（全件 renumber 不要）。キーは同一 `status` 内で昇順ソートすることで表示順を決定する。

## 8. 技術スタック

| 区分 | 採用技術 | 理由 |
|:---|:---|:---|
| ランタイム | Bun | 高速・軽量。`bun:sqlite` ビルトイン対応でサードパーティドライバー不要。 |
| フレームワーク | Next.js 15 (App Router) | フロント / API を一体化。Server Actions で DB 操作を簡潔に記述。 |
| UI | Tailwind CSS + shadcn/ui | コンポーネントベースで洗練されたデザインを低コストで実現。 |
| DB | SQLite + **Drizzle ORM** | ファイルベースで環境構築不要。Drizzle は `bun:sqlite` と1:1対応し TypeScript のみで完結。Prisma のような外部バイナリが不要。 |
| DnD | **@atlaskit/pragmatic-drag-and-drop** | コア 4.7 kB・フレームワーク非依存。Jira/Confluence 本番採用済みで信頼性が高く、タスク増加時の描画モタつきがない。 |
| 並び順管理 | **fractional-indexing** | DnD 並び替え時に移動タスク1件のみ更新。全件 renumber 不要で DB 負荷最小。週22万 DL の実績ある軽量ライブラリ。 |
| バリデーション | valibot | 軽量・型安全。Server Actions のフォーム検証に使用。 |
| URL 状態管理 | nuqs | フィルター状態を URL に同期。 |

### 注意事項：SQLite ファイルの管理

- `.gitignore` に `*.db` / `*.sqlite` を必ず追加し、DB ファイルをリポジトリに含めない。

```

│  タスク管理             [+ タスク追加]  [メンバー管理]  │

│ │ 担当者アイコン│ │ │ …        │ │ │ …                  │ │
│ │ ● 高       │ │ └──────────┘ │ └────────────────────┘ │
│ └───────────┘ │              │                        │
└───────────────┴──────────────┴────────────────────────┘
```

### タスクカード
