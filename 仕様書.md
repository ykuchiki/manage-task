|:---|:---|

---

## 1. 目的・背景

チームメンバーが「誰が」「何を」「どの程度の優先度で」抱えているかを瞬時に把握できる社内向けツールを構築する。これにより、リソース配分の最適化・業務の透明性向上・マネジメントコストの低減を実現する。

**設計方針:** 学習コストゼロ。入力ハードルを極限まで下げ、日常的に使い続けられるシンプルさを最優先とする。

---

## 2. スコープ

### 対象

- 社内チームメンバーのタスク管理
- マネージャーによるチーム全体のタスク俯瞰

### 対象外

- 外部公開・認証機能（ローカル運用のため不要）
- 工数管理・時間追跡
- 外部サービス連携（Slack / GitHub 等）

---

## 3. 実装の流れ

仕様を実装する際の推奨順序。各フェーズが完了してから次に進む。

### Phase 1: 基盤

| 順 | 作業 | 成果物 | 検証 |
|:---|:---|:---|:---|
| 1.1 | Next.js + Bun でプロジェクト初期化 | `package.json`, `next.config.ts` | `bun dev` で起動 |
| 1.2 | Drizzle + SQLite 接続、スキーマ定義 | `_db/schema.ts`, `drizzle.config.ts` | `drizzle-kit push` でテーブル作成 |
| 1.3 | マイグレーション実行、DB 初期接続確認 | `drizzle/` 配下の SQL | 起動時エラーなし |
| 1.4 | メンバー管理画面の骨組み | `/members` ページ | 空の一覧が表示される |

### Phase 2: メンバー管理

| 順 | 作業 | 成果物 | 検証 |
|:---|:---|:---|:---|
| 2.1 | users の CRUD（queries / mutations / Server Actions） | `_db/queries.ts`, `_db/mutations.ts`, actions | 作成・一覧・編集・削除が動く |
| 2.2 | メンバー一覧 UI（追加フォーム・編集・削除ボタン） | `/members` の完成 | M-01, M-02, M-03 達成。**6-11（メンバー追加・編集・削除の画面更新）** を満たすこと。 |
| 2.3 | メンバー削除時のタスク `user_id` を NULL に更新 | mutations に連携処理 | 削除後もタスクが残る。ダッシュボードの該当タスクは「未割当」表示になる。 |

### Phase 3: タスク管理（CRUD のみ）

| 順 | 作業 | 成果物 | 検証 |
|:---|:---|:---|:---|
| 3.1 | tasks の CRUD（position の初期値を fractional-indexing で生成） | queries / mutations / actions | タスクの作成・一覧・編集・削除 |
| 3.2 | ダッシュボードの静的カンバン（列＝ステータス、カードは表示のみ） | `/` ページ | タスクがステータス別に表示される |
| 3.3 | タスク追加モーダル（担当者ドロップダウン＝メンバー一覧） | ダッシュボードの「+ タスク追加」 | T-01 達成。**6-11（タスク追加の画面更新）** を満たすこと。 |
| 3.4 | タスク編集・削除（カードクリックで編集、削除は確認付き） | タスクカードの操作 | T-03, T-04 達成。**6-11（タスク編集・削除の画面更新）** を満たすこと。 |

### Phase 4: DnD とフィルター

| 順 | 作業 | 成果物 | 検証 |
|:---|:---|:---|:---|
| 4.1 | pragmatic-drag-and-drop で列間・列内の DnD | カンバンの DnD | T-05, T-07 達成。**6-11（DnD の画面更新要件）** を満たすこと。 |
| 4.2 | DnD 時に Server Action で status / position を更新 | mutations | リロード後も並びが保持される。失敗時はカードを元の位置に戻す。 |
| 4.3 | nuqs で担当者・優先度フィルターを URL に紐付け | フィルタバー | D-02, D-03, D-04 達成。**6-11（フィルター変更）** を満たすこと。 |
| 4.4 | 優先度のワンクリック変更 UI | タスクカード | T-06 達成。**6-11（優先度変更）** を満たすこと。 |

### 依存関係

```
Phase 1 ──► Phase 2 ──► Phase 3 ──► Phase 4
   │            │            │
   └─ DB 必須   └─ メンバー   └─ タスク CRUD
                    必須           必須
```

- Phase 2 完了前に Phase 3 の「担当者ドロップダウン」は空になる（メンバー 0 件でもタスクは「担当者なし」で作成可能とする）。
- タスクの position は Phase 3 で初期値生成、Phase 4 で DnD による更新を実装する。

## 6. 要件の詳細仕様（網羅）

### 6-1. 入力バリデーション

| 項目 | ルール | エラー時の表示 |
|:---|:---|:---|
| タスク名 | 1〜100文字。前後の空白はトリム。空文字は不可。 | 「タスク名は1〜100文字で入力してください」 |
| メンバー名 | 1〜50文字。前後の空白はトリム。空文字は不可。 | 「名前は1〜50文字で入力してください」 |
| アバター色 | HEX 形式（例: `#3b82f6`）。事前定義の選択肢から選ぶ。 | 選択肢のみ許可（不正値は送信されない） |
| 担当者 | users.id のいずれか、または NULL（担当者なし） | ドロップダウンに「未割当」を用意 |
| 優先度 | 1（高）/ 2（中）/ 3（低） | 選択肢のみ |
| ステータス | 0（未着手）/ 1（進行中）/ 2（完了） | 選択肢のみ |

### 6-2. 空状態

| 状態 | 表示 |
|:---|:---|
| メンバー 0 件 | 「メンバーがいません。追加してください。」+ 追加ボタン |
| タスク 0 件（ダッシュボード全体） | 各列に「タスクがありません」+ その列への追加誘導 |
| タスク 0 件（フィルター適用後） | 該当するタスクがない旨を表示 |
| 担当者 0 件時のタスク追加 | 担当者ドロップダウンに「未割当」のみ。タスクは作成可能。 |

### 6-3. 削除時の挙動

| 対象 | 挙動 |
|:---|:---|
| タスク削除 | 確認ダイアログ「このタスクを削除しますか？」→ OK で削除 |
| メンバー削除 | 確認ダイアログ「○○を削除しますか？紐づくタスクは未割当になります。」→ OK で削除 |

### 6-4. エラー処理

| 事象 | 挙動 |
|:---|:---|
| バリデーションエラー | フォーム直下にエラーメッセージ表示。Server Action は `{ success: false, error: string }` を返す。 |
| DB 接続失敗 | 画面に「データベースに接続できません」を表示。 |
| 存在しないタスク/メンバーの編集 | 404 相当。一覧に戻す or トーストでエラー通知。 |
| 並行更新（同じタスクを複数人が編集） | MVP では考慮しない。後で競合検知を検討。 |

### 6-5. ローディング・フィードバック

| 操作 | 挙動 |
|:---|:---|
| ページ初回読み込み | スケルトン or ローディング表示 |
| Server Action 実行中 | ボタンの disabled + スピナー等 |
| Server Action 成功後 | `revalidatePath` で即時反映。成功トーストは不要（変更が画面に出れば十分）。 |
| Server Action 失敗後 | エラーメッセージをフォーム近くに表示 |

### 6-6. フィルター仕様

| フィルター | 選択肢 | デフォルト |
|:---|:---|:---|
| 担当者 | 全員 + 各メンバー | 全員 |
| 優先度 | 全て + 高 / 中 / 低 | 全て |

「全員」「全て」はフィルターなし（全件表示）。

### 6-7. タスク追加 UI

- ダッシュボード右上の「+ タスク追加」クリックでモーダル（または Dialog）を開く。
- 期限・説明欄は入力ハードルを下げるため MVP では設けない。
- フォーム項目: タスク名、担当者、優先度、ステータス（初期値: 未着手）。
- 新規タスクの `position` は、選択したステータス列の末尾に `generateKeyBetween(lastPosition, null)` で生成する。

### 6-8. タスク編集 UI

- タスクカードをクリックで編集モーダルを開く。またはカード上に編集アイコンを置く。
- 編集可能項目: タスク名、担当者、優先度（ステータスは DnD で変更するため編集フォームには含めない想定、もしくは両方対応可）。

### 6-9. メンバー管理 UI

- `/members` で一覧表示。各行に編集・削除ボタン。
- 追加はページ上部にフォーム、または「メンバーを追加」ボタンでモーダルを開く。
- 編集はモーダルまたはインライン編集。どちらでも可（実装時に選ぶ）。

### 6-10. 初期データ

- DB 初回作成時は空。シードデータは任意（開発用に `drizzle-kit seed` 等で用意する場合は `scripts/seed.ts` を参照）。

### 6-11. 操作と画面更新の対応（必須）

各操作後に画面がどう更新されるかを厳密に定義する。設計・実装の受け入れ基準となる。

#### タスクの DnD（列間・列内）

| 項目 | 要件 |
|:---|:---|
| トリガー | ユーザーがカードをドロップした瞬間 |
| 即時フィードバック | ドロップ直後にカードを新しい位置に表示する（オプティミスティック UI）。ドラッグ中の「浮き」表現は解除し、カードはドロップ先に固定される。 |
| サーバー処理 | ドロップと同時に Server Action を呼び、DB の `status` と `position` を更新する。 |
| 成功時 | カードは既に正しい位置にある。`revalidatePath('/')` でサーバーコンポーネントを再取得し、表示と DB の整合を保つ。フラッシュ・二重表示・カードの飛び跳ねは発生させない。 |
| 失敗時 | カードを**元の列・元の位置に戻す**。エラーメッセージをトーストまたは画面内に表示する。ユーザーは「保存されなかった」と分かること。 |
| リロード後 | ページを再読み込みしても、DnD 後の並び・列がそのまま表示されること。 |

#### タスク追加

| 項目 | 要件 |
|:---|:---|
| トリガー | フォームの「追加」ボタン押下 |
| 成功時 | モーダルを閉じる。新規タスクが**指定したステータス列の末尾**に表示される。`revalidatePath('/')` で一覧を再取得。 |
| 失敗時 | モーダルは開いたまま。フォーム直下にエラーメッセージを表示。入力内容は保持する。 |

#### タスク編集

| 項目 | 要件 |
|:---|:---|
| トリガー | 編集モーダルの「保存」ボタン押下 |
| 成功時 | モーダルを閉じる。該当カードの**タスク名・担当者・優先度**が即時更新される。カードの列・並び順は変わらない。`revalidatePath('/')` で再取得。 |
| 失敗時 | モーダルは開いたまま。フォーム直下にエラーメッセージを表示。 |

#### タスク削除

| 項目 | 要件 |
|:---|:---|
| トリガー | 確認ダイアログの「削除」押下 |
| 成功時 | ダイアログを閉じる。該当カードが**一覧から即座に消える**。`revalidatePath('/')` で再取得。 |
| 失敗時 | ダイアログは開いたまま。エラーメッセージを表示。 |

#### メンバー追加

| 項目 | 要件 |
|:---|:---|
| トリガー | フォーム送信 |
| 成功時 | 一覧に新規メンバーが**即座に追加**される。フォームはクリアするか、追加用フォームを閉じる。`revalidatePath('/members')`。 |
| 失敗時 | フォームにエラーメッセージを表示。入力内容は保持。 |

#### メンバー編集

| 項目 | 要件 |
|:---|:---|
| トリガー | 編集フォームの保存 |
| 成功時 | 一覧の該当行が**即座に更新**される。ダッシュボード（`/`）を開いている場合、該当メンバーを担当者とするタスクカードのアイコン・表示名も次回表示時に更新される（`revalidatePath` で `/` も再検証するか、メンバー一覧とタスク一覧で同一データを参照する設計とする）。 |
| 失敗時 | フォームにエラーメッセージを表示。 |

#### メンバー削除

| 項目 | 要件 |
|:---|:---|
| トリガー | 確認ダイアログの「削除」押下 |
| 成功時 | ダイアログを閉じる。一覧から該当メンバーが**消える**。ダッシュボード上でそのメンバーが担当だったタスクは**「未割当」表示**に変わる。`revalidatePath` で `/members` と `/` の両方を再検証。 |
| 失敗時 | ダイアログは開いたまま。エラーメッセージを表示。 |

#### フィルター変更

| 項目 | 要件 |
|:---|:---|
| トリガー | 担当者または優先度の Select を変更 |
| 即時反映 | 表示タスクが**即座に**フィルタ条件に合わせて切り替わる。サーバー再取得は不要（クライアントでフィルタする場合）または nuqs で URL が変わり、サーバーコンポーネントが再レンダーされる設計とする。 |
| URL 同期 | `?assignee=xxx&priority=high` 等、フィルター状態が URL に反映される。 |

#### 優先度のワンクリック変更（T-06）

| 項目 | 要件 |
|:---|:---|
| トリガー | タスクカード上の優先度バッジをクリック（またはドロップダウンで選択） |
| 成功時 | カード上の優先度バッジが**即座に**新しい優先度の色・文言に変わる。`revalidatePath('/')` で再取得。 |
| 失敗時 | バッジは元のまま。エラーメッセージを表示。 |

#### 共通原則

- **一貫性**: どの操作でも「操作 → サーバー反映 → 画面更新」の流れが途切れないこと。
- **二重送信防止**: Server Action 実行中は送信ボタンを `disabled` にし、連打で複数回送信されないようにする。
- **複数タブ**: MVP では考慮しない。別タブで変更した内容は、そのタブをリロードするまで反映されなくてよい。

## 8. データモデル

シンプルさを保つためテーブルは 2 つのみとする。

### users

| カラム | 型 | 制約 | 説明 |
|:---|:---|:---|:---|
| id | TEXT | PK | cuid2 |
| name | TEXT | NOT NULL | 表示名（1〜50文字） |
| avatar_color | TEXT | NOT NULL | UI アイコン色（HEX） |
| created_at | DATETIME | NOT NULL | 作成日時 |

### tasks

| カラム | 型 | 制約 | 説明 |
|:---|:---|:---|:---|
| id | TEXT | PK | cuid2 |
| title | TEXT | NOT NULL | タスク名（1〜100文字） |
| user_id | TEXT | FK → users.id, NULL 可 | 担当者（NULL = 未割当） |
| priority | INTEGER | NOT NULL | 1=高 / 2=中 / 3=低 |
| status | INTEGER | NOT NULL | 0=未着手 / 1=進行中 / 2=完了 |
| position | TEXT | NOT NULL | 並び順キー（Fractional Indexing） |
| created_at | DATETIME | NOT NULL | 作成日時 |
| updated_at | DATETIME | NOT NULL | 最終更新日時 |

**並び順の設計方針（T-07）:**  
`position` には `fractional-indexing` の `generateKeyBetween(prev, next)` で生成した文字列を格納する。DnD 時は移動したタスク 1 件のみ `position` を更新。同一 `status` 内で `position` 昇順ソートで表示順を決定する。

## 10. 技術スタック

| 区分 | 採用技術 | 理由 |
|:---|:---|:---|
| ランタイム | Bun | `bun:sqlite` ビルトイン対応。サードパーティドライバー不要。 |
| フレームワーク | Next.js 15 (App Router) | フロント / API 一体化。Server Actions で DB 操作を簡潔に記述。 |
| UI | Tailwind CSS + shadcn/ui | コンポーネントベースで洗練された UI を低コストで実現。 |
| DB | SQLite + **Drizzle ORM** | ファイルベースで環境構築不要。`bun:sqlite` と 1:1 対応。TypeScript のみで完結。 |
| DnD | **@atlaskit/pragmatic-drag-and-drop** | コア 4.7 kB。Jira/Confluence 本番採用済み。タスク増加時の描画モタつきなし。 |
| 並び順管理 | **fractional-indexing** | DnD 時は移動タスク 1 件のみ更新。全件 renumber 不要。 |
| バリデーション | valibot | 軽量・型安全。Server Actions のフォーム検証に使用。 |
| URL 状態管理 | nuqs | フィルター状態を URL に同期。 |

### 注意事項：SQLite ファイルの管理

- `.gitignore` に `*.db` / `*.sqlite` を必ず追加する。
- データ消失リスクに備え、1 日 1 回 DB を別ディレクトリへコピーする cron スクリプトを用意する（例: `scripts/backup.sh`）。

```
┌──────────────────────────────────────────────────────┐
│  タスク管理             [+ タスク追加]  [メンバー管理]  │
│  担当者: [全員 ▼]  優先度: [全て ▼]                    │
├───────────────┬──────────────┬────────────────────────┤
│   未着手       │   進行中      │   完了                 │
│ ┌───────────┐ │ ┌──────────┐ │ ┌────────────────────┐ │
│ │ タスクカード│ │ │タスクカード│ │ │ タスクカード        │ │
│ │ 担当者アイコン│ │ │ …        │ │ │ …                  │ │
│ │ ● 高       │ │ └──────────┘ │ └────────────────────┘ │
│ └───────────┘ │              │                        │
└───────────────┴──────────────┴────────────────────────┘
```

### タスクカード

- タスク名（1〜2行）
- 担当者アイコン（アバター色のイニシャル円。未割当時は「-」等）
- 優先度バッジ（高=赤 / 中=黄 / 低=緑）

---

## 12. デザイン仕様

実装時に参照するビジュアル・インタラクションの指針。shadcn/ui をベースに、Tailwind でカスタマイズする。

### 12-1. カラーパレット

| 用途 | 指定 | 備考 |
|:---|:---|:---|
| 背景（メイン） | `slate-50` / `zinc-50` | 控えめな白に近い色 |
| 背景（カード） | `white` | カード・モーダルは白で浮かせる |
| テキスト（メイン） | `slate-900` / `zinc-900` | 見出し・本文 |
| テキスト（サブ） | `slate-500` / `zinc-500` | 補足・ラベル |
| アクセント | `primary`（shadcn テーマ） | 主要ボタン・リンク |
| ボーダー | `slate-200` / `zinc-200` | カード・列の区切り |

**優先度バッジ:**

| 優先度 | 背景色 | テキスト |
|:---|:---|:---|
| 高 | `red-100` | `red-800` |
| 中 | `amber-100` | `amber-800` |
| 低 | `emerald-100` | `emerald-800` |

**アバター色（事前定義）:**  
`#ef4444`, `#f97316`, `#eab308`, `#22c55e`, `#3b82f6`, `#8b5cf6`, `#ec4899`, `#6b7280` から選択。

### 12-2. タイポグラフィ

| 要素 | 指定 | 用途 |
|:---|:---|:---|
| ページタイトル | `text-xl font-semibold` | 「タスク管理」等 |
| セクション見出し | `text-sm font-medium text-muted-foreground` | カンバン列名「未着手」等 |
| タスク名 | `text-sm font-medium` | カード内のタスクタイトル |
| ラベル・補足 | `text-xs text-muted-foreground` | フォームラベル、アイコン横のテキスト |
| ボタン文言 | `text-sm` | 統一 |

システムフォント（`ui-sans-serif`）を基本。必要に応じて Noto Sans JP 等を追加。

### 12-3. 余白・レイアウト

| 要素 | 指定 |
|:---|:---|
| ページ外側 | `p-4` または `p-6`（画面幅に応じて max-w で中央寄せ） |
| カンバン列間 | `gap-4` |
| カード間（列内） | `gap-2` |
| カード内パディング | `p-3` |
| フォーム項目間 | `space-y-4` |
| ボタンと入力の間 | `gap-2` |

### 12-4. カンバン列

- 列幅: `min-w-[280px]`（タスクカードが読みやすい最小幅）
- 列背景: `bg-slate-100/50` または `bg-zinc-100/50`（メイン背景よりやや濃い）
- 列間: 縦線または `rounded-lg` + `border` で区切り
- 列ヘッダー: ステータス名 + その列のタスク数（任意）

### 12-5. タスクカード

- 最小高さ: 約 72px。コンテンツに応じて伸びる。
- 背景: `white`、`rounded-md`、`border border-slate-200`
- ホバー: `hover:shadow-sm hover:border-slate-300`（クリック可能であることが分かる）
- DnD ドラッグ中: `opacity-90`、`shadow-lg`、`rotate-1` 程度で「持ち上げ」感を出す
- DnD ドロップゾーン: ドロップ可能な列・位置に `ring-2 ring-primary/30` でハイライト（任意）

**カード内レイアウト（上から順）:**

1. タスク名（1〜2行、`line-clamp-2`）
2. フッター行: 担当者アイコン（左）+ 優先度バッジ（右）

### 12-6. ボタン

- 主要アクション（タスク追加、保存）: `Button` variant="default"
- 二次アクション（キャンセル、閉じる）: `Button` variant="outline"
- 破壊的操作（削除）: `Button` variant="destructive"
- アイコン + 文言は `gap-2` で間隔

### 12-7. モーダル・ダイアログ

- shadcn の `Dialog` を使用
- 幅: `max-w-md`（タスク追加・編集）、`max-w-lg`（メンバー編集など項目が多い場合）
- ヘッダー: タイトル + 閉じるボタン
- フッター: キャンセル（左）+ 確定（右）

### 12-8. フィルターバー

- 担当者・優先度は `Select` コンポーネント
- ラベルと Select を横並び、`gap-2`。複数 Select の間は `gap-4`
- 高さ: 他コンテンツと揃え、詰まらずに並べる

### 12-9. 空状態

- 中央寄せのアイコン + 文言 + アクションボタン
- アイコンは `LucideReact` の `Inbox`（タスクなし）、`Users`（メンバーなし）等
- 文言は `text-muted-foreground` で控えめに

### 12-10. インタラクション・トランジション

| 操作 | 挙動 |
|:---|:---|
| モーダル開閉 | `Dialog` のデフォルトアニメーション（フェード + スケール） |
| ボタンホバー | 軽い背景色変化（shadcn デフォルト） |
| フォーカス | `ring-2 ring-ring ring-offset-2`（アクセシビリティのため必須） |
| DnD ドラッグ開始 | 0.15s 程度でカードが「浮く」感じ。`transition-shadow` 等 |
| フィルター切替 | 即時反映。ローディングはデータ量次第で検討 |

### 12-11. レスポンシブ

- 主用途: デスクトップ（社内 PC 前提）
- 最小幅: 約 1024px でカンバン 3 列が横並びになることを想定
- それ未満: 列を横スクロールにするか、1 列ずつ縦積み（実装時に決定）
